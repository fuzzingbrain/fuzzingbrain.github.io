Most reports of models finding bugs count a crash. A crash is not a solve. FuzzingBrain Bench is built so the program itself decides whether a model found the exact documented bug or merely tripped some other fault. Three design choices make that possible.

🔒

Answer-free by construction

Every challenge is a sealed Docker image named by a neutral alias. No patch, no fix commit, no target line. The PoC and the expected fault live only behind a remote oracle, so a model cannot read the answer off disk.

⚖️

A deterministic oracle

No LLM judge. The candidate input runs through the real sanitizer-instrumented harness on a private grading service, which returns only a verdict. The same input always grades the same way.

🪜

A capability ladder

Grading is five nested levels from reach to crash to differential to class to site. A model earns partial credit for getting close, and a full solve only when it lands the exact documented fault at the exact place.

Run it in five minutes

Only Docker and a model key are needed. No build, no answer key, nothing to download but the sealed image.

# install
git clone https://github.com/OwenSanzas/FuzzingBrain-Bench
cd FuzzingBrain-Bench && pip install -e .

# put your key in ./.env, then run one challenge
fb-bench run avro-03 --model claude-haiku-4-5

# or sweep the whole corpus with one model
python -m fbbench.sweep.orchestrator --models gpt-5.5 --bugs all --exp run1

The ladder separates two skills

Reaching the buggy code is navigation. Turning that into the exact crash is exploitation. The gap between the two is where models differ most.

A crash is not a solve

The oracle fires the crash level for any fault, but class and site require the documented one. That gap is why crash-counting overstates real skill.

Of every run that crashed the harness

Only three quarters of crashes were the preset bug. Grading on did-it-crash alone would over-count real solves by about a quarter.

right class and site right class, wrong site a different bug

Why strict grading matters

  • 64% of runs produced some crash, but many were the wrong fault.
  • The class and site levels cleanly split found a bug from found the bug.
  • The weaker model goes off-target more often, so it thinks it won when it did not.
  • A deterministic oracle makes this call the same way every single time.

The bottom line

Difficulty here is not about how deep a bug sits or how many bytes the input needs. It is about constructing the one input that satisfies the crash condition. The best frontier model recovers most bugs cold, yet a hard tail resists every model in every mode. That is the frontier this benchmark measures.
