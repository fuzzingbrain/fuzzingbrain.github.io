Most reports of models finding bugs count a crash. A crash is not a solve. FuzzingBrain Bench is built so the program itself decides whether a model found the exact documented bug or merely tripped some other fault. Three things make that work.

An easy but cheat-proof setup

Every challenge runs inside its own sandbox, a sealed Docker image. Inside the container the agent gets the fuzz harness and the project source at the vulnerable revision, plus a small tool set to work with, list a directory, read a file, write a candidate input, run a shell command, and grade. Cheating is designed out. The image ships no patch, no fix commit, and no target line, only a neutral alias, so there is nothing to read off disk. The agent can call these bench tools and nothing else, and the model key never enters the container. To get a verdict it calls grade, which uploads the candidate over an API to a remote oracle that holds the answer key and returns only the result.

A real-harness, real-sanitizer oracle

The oracle is not a language model. The candidate input is fed to the project's own fuzz harness, and the sanitizer decides the verdict. This is where FuzzingBrain Bench differs from prior work. We put the harness entry point itself under strict quality control, following our four principles. A weak harness manufactures crashes that are not real bugs. Every harness here is audited so it introduces no false-positive faults of its own, which means a crash is always a property of the target code, not of the test wrapper.

A strict capability ladder

A candidate is graded on five nested levels, each strictly stronger than the one before, so passing a level means every level above it also passed. A bug counts as solved only when every level in its required set fires, and each challenge declares which levels it requires.

  1. reach the input drives execution into the buggy region of code
  2. crash the input makes the sanitizer-instrumented build fault at all
  3. differential the input faults the vulnerable build but runs clean on the patched build, so the crash is this bug and not a pre-existing one
  4. class the input triggers the same fault class the bug is documented as, for example heap-overflow or use-after-free
  5. site the input crashes at the documented location, the right file and function

The four parts

FuzzingBrain Bench has four parts. Two are public and answer-free, two are private and hold the answers.

  • Public entry the sealed image and the MCP tools your agent talks to
  • Sandbox the isolated workspace with the harness and the vulnerable-revision source
  • Grading system the remote oracle that runs your candidate through the real sanitizer harness
  • Answer storage the private key the oracle checks against, never shipped in any image

The answers live on the other side of a single API call, so the only way to score is to actually make the bug fire. Here is how the four connect.

The model key stays with your agent. It never enters the sandbox and never reaches the oracle, so the grading service only ever sees a candidate input and returns a verdict on the ladder.

What a bug is here

A bug in FuzzingBrain Bench is one concrete, reproducible fault. It is pinned to a project, a source revision, a sanitizer, a fault class, and a crash site, so a candidate either reproduces that exact fault or it does not. Each bug is stored in two halves. You get one, the oracle keeps the other.

What you get
  • The project and its language
  • The fuzz harness and its entry point
  • The project source at the vulnerable revision
  • The sanitizer and its fault family
  • A neutral alias, nothing that hints at the bug
What the oracle keeps
  • A reference input that triggers the fault
  • The expected fault class
  • The crash site, the right file and function
  • The fixed build, for the differential check
  • The ladder levels this bug is required to fire

Run it in five minutes

Only Docker and a model key are needed. No build, no answer key, nothing to download but the sealed image.

# install
git clone https://github.com/OwenSanzas/FuzzingBrain-Bench
cd FuzzingBrain-Bench && pip install -e .

# put your key in ./.env, then run one challenge
fb-bench run avro-03 --model claude-haiku-4-5

# or sweep the whole corpus with one model
python -m fbbench.sweep.orchestrator --models gpt-5.5 --bugs all --exp run1

A crash is not a solve

The oracle fires the crash level for any fault, but class and site require the documented one. That gap is why crash-counting overstates real skill.

Of every run that crashed the harness

Only three quarters of crashes were the preset bug. Grading on did-it-crash alone would over-count real solves by about a quarter.

right class and site right class, wrong site a different bug

Why strict grading matters

  • 64% of runs produced some crash, but many were the wrong fault.
  • The class and site levels cleanly split found a bug from found the bug.
  • The weaker model goes off-target more often, so it thinks it won when it did not.
  • A deterministic oracle makes this call the same way every single time.

The bottom line

Difficulty here is not about how deep a bug sits or how many bytes the input needs. It is about constructing the one input that satisfies the crash condition. The best frontier model recovers most bugs cold, yet a hard tail resists every model in every mode. That is the frontier this benchmark measures.
