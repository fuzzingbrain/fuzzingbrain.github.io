基于四原则的对抗性生成 Harness:470 个 Harness × 30 个漏洞 × 2 周

上一篇把四原则当 审计 准则、事后接住 OSS-Fuzz 现有的 586 个 harness。这一篇反过来 —— 把它当 生成 准则:从 P4 选攻击面开始,配合 Logic Group 切分语义单元,再用 Stage 4 对抗性 probe 当出厂 QA。两周时间为 Chromium 及其上游依赖部署 481 个 harness(头条数字四舍五入到 470),提了 近 30 个对维护者形成事实的真漏洞(精确是 28 = 7 fixed + 9 confirmed + 12 submitted,14 High / 13 Medium / 1 Low),另有 19 个 finding 还在 file 队列,只有 2 个 dropped。

背景:四原则只是审计工具吗?

上一篇把四原则(P1 Logic Correctness / P2 API Protocol Compliance / P3 Security Boundary Respect / P4 Entry Point Adequacy)当成事后审计的工具 —— harness 已经在 OSS-Fuzz 上跑着了,我们逐条检查它对不对。结果是 586 个 harness 修了 53 个、拦下 14 个 FP、顺手挖出一个潜伏 25 年的 OpenSSL 老 bug。四条原则的源码层定义和 16 条子检查清单(P1.1–P1.8 / P2.1–P2.8)都在那一篇里,本文不再重复。

但同样四条规则也可以在 harness 还没写之前就用上 —— 把"什么样的 harness 才算合格"的标准内化到生成阶段,而不是事后接住。这一篇讲的就是这件事:把四原则反过来用(P4 → P3 → P2 → P1),配合一个把库切成可独立 fuzz 的语义单元的工具 —— Logic Group,在两周内为 Chromium 及其上游依赖生成、部署、跑了 481 个 fuzzer(标题里的 470 是头条数字,这是当前精确值)。

为什么生成时要"反过来"用

Part 1 的审计只机械跑了 P1.1–P1.8 + P2.1–P2.8 共 16 条子检查。P3 / P4 没进静态过 —— 因为 harness 已经在线上跑了好几年,入口是上游 reviewer 当年挑的,我们事后只在 borderline case 上人工核一下;P3 / P4 本身就是判断题,不是机械题。这是"接住已有 harness"这个出发点带来的奢侈。

生成阶段没有这个奢侈。入口是我们当场挑的 —— 没人替我们做 P4(目标在不在真攻击面上)和 P3(能不能从公开侧复现)。如果跳过这两步、先让 LLM 写代码,等 60 秒 fuzz 跑完再回头看"入口选错了",前面的所有工作全要扔。所以反向不是"颠倒一下次序",是补上审计时不必做、生成时必须做、且必须做在所有别的事情之前的两步:

  • P4 选攻击面 —— 公开 API 闭包里要有内存安全相关操作,否则换目标
  • P3 验边界 —— 入口尽量公开;若用内部符号,公开侧必须能复现
  • P2 研协议 —— 摸清调用顺序 / 生命周期 / 参数约束 / 配对清理(写 harness 之前的功课)
  • P1 写代码 —— 前三层都对了,P1 就是孤立的工程问题,直接套 16 条子检查

任何一步失败都不进入下一步。对 LLM 来说,这把"自由发挥的空间"逐层收窄 —— 产出的 harness 不会跑题到内部 helper 上,也不会用错误的调用顺序把库的真实行为遮掉。

还有一条工程边界值得点出:P3 / P4 的判断不交给 LLM 单点决策。"这个用法是否合理 / 这个入口是否在真实攻击面上"是项目语境强相关的判断 —— 资深 reviewer 凭项目经验给得出,没有项目上下文的第三方通用 LLM 给不出。所以反向流水线在 Stage 1(P4/P3)走的是"LLM 列候选 + 静态调用图给可达性 + 资深 reviewer 在 borderline case review"的组合,LLM 是工具不是裁判。这条边界让 481 个 fuzzer 在质量上能被人问责。

Logic Group:一个库的"可 fuzz 语义单元"

P4 让你选攻击面,但攻击面这个词太抽象。一个真实的 C/C++ 库往往导出几百到上千个公开符号,你不可能给每个函数都写一个 harness —— 那样既低效,又会让 fuzzer 在意义不大的 wrapper 上浪费 CPU。

我们引入的概念叫 Logic Group(LG) —— 一个库被切分成的最小语义单元,每个 LG 由四个字段定义:

name
语义标签
"parse PNG image" 这种描述
E
Entry set
接收 fuzz bytes 的公开 API
C
Core set
从 E 可达的核心实现函数
desc
攻击面描述
"为什么这个 LG 值得 fuzz"

一个 LG 对应一个 fuzzer。LG 之间不重叠 —— 如果两个 LG 的 core set 大量交集,我们会合并;如果一个候选 LG 不能覆盖任何内存安全相关的代码,我们会丢弃。每个库通常被切成 5–10 个 LG,具体根据公开 API 拓扑决定。

举一个具体例子。openscreen 是 Chromium 的 Cast streaming 实现,我们对它切出来的 LG 包括:

  • openscreen_receiver_message_json_parse — 解析 receiver 端发回来的 RECEIVER_MESSAGE JSON 体
  • openscreen_sender_message_json_parse — 解析 sender 发出的 SENDER_MESSAGE JSON 体
  • openscreen_answer_messages_deep_substructs — 解析 ANSWER 消息里的嵌套子结构(AudioConstraints / VideoConstraints / ...)
  • openscreen_offer_messages_round_trip — OFFER 的解析-序列化往返
  • ...

每个 LG 是一个独立的 fuzz 入口,对应一个独立的 harness。这样切的好处稍后会在结果里看到:同一类 jsoncpp 滥用 bug 在三个不同的 LG 里被独立触发,每个 LG 都给上游一个干净的、不互相牵连的报告。

四阶段流水线:LG → API → 代码 → 验证

反向四原则在工程化上落成四个阶段。每个阶段的产物是下一个阶段的输入,任何阶段不通过就直接退回 LLM 重做。

Stage 1 — Logic Group Discovery(对应 P4 / P3)

一个 LLM agent 读项目源码 + 公开 header,枚举 5–10 个候选 LG,然后用静态调用图给每个 LG 算 danger score: 

danger(LG) = sum_over_g_in_C( unsafe_ops_in(g) / depth_from_E_to_g )

unsafe_ops 包括 memcpy、malloc、free、strcpy、sprintf 等。深度折扣 1/d 让排名集中在"从公开入口直接可达的攻击面",不会被深处的库内部细节吞掉。前 5 个 LG 进入下一阶段;P3 验证(可达性必须从公开 API 出发)在这一阶段同步过完。

Stage 2 — API Protocol Research(对应 P2)

选定 LG 之后,另一个 agent 读对应入口和 core 函数的头文件 + 注释 + 关键调用点的源码,给每个 entry function 整理出一份协议文档:

  • 调用顺序约束(必须先 init 再 use 再 destroy)
  • 对象生命周期(谁分配、谁释放、能不能跨调用复用)
  • 参数合法范围(int 的有效区间、enum 的合法值集)
  • 返回值语义(NULL on failure / negative error code / out-param)
  • 必须配对调用的清理 API

这份文档每条都必须落到 file:line,不允许 LLM 自由发挥。这是"P2 写在 harness 之前"的关键 —— 等 harness 写完再回头审,LLM 已经被自己刚写的代码锚定了,改不动。先把协议研究清楚,再写代码。

Stage 3 — Static-Driven Build(对应 P1 / P2)

harness-generator agent 拿到 P2 文档之后写 harness 源码 —— 一次写完整,不是逐行试错。然后跑一个有上限的 build-fix loop:编译失败 → 把错误回喂给 LLM → 让它修 → 再编 → 直到通过或者次数耗尽。

和"写一个空壳让 LLM 填"的范式不同,我们让 LLM 一次性产出整个 LLVMFuzzerTestOneInput 加配套 setup / teardown,这样它能跨函数推理 —— LLM 在这种长跨度任务上的表现远好于 wrapper 式的逐函数生成。

Stage 4 — Adversarial Validation(对应 P1 / P2)

最后一道闸门是 adversarial probing。系统对刚生成的 harness 主动构造对抗输入,跑两类探针:

  • Reach probe —— 在 GDB 里给目标 API 下断点,跑 harness 看断点能不能命中。命不中就说明 fuzz bytes 没流到目标,P1 失败。
  • Run probe —— libFuzzer + ASan + LSan,跑短时长。任何 crash 立刻路由到 P1.1–P1.8 / P2.1–P2.8 的 16 个子检查里做 triage。如果 crash 归到 harness 自身违规,直接打回让 LLM 重写;只有归到库的真行为差异才算可能的 bug 候选。

前一类负责"harness 真的在测库",后一类负责"harness 自己不会自爆"。两类都过才会把 harness 标记为 ready-for-discovery。

实测:两周 / 481 harness / 58 个库

这套流水线我们从 2026 年 4 月中旬开始针对 Chromium 及其上游依赖部署。约两周时间,生成、构建、并跑过对抗性验证的 harness 数量是 481 个,覆盖 58 个库。pipeline 抛出的 finding 共 49 个;按"对维护者形成事实"这条线切,可以分三段看:

  • 28 个对维护者形成事实的漏洞(标题里说的"近 30"):7 fixed(已 patch 入主仓)+ 9 confirmed(维护者 ack)+ 12 submitted(已发 issue/PR、等回复)。
  • 19 个 recorded:本地 verify 已过、还在 file 队列里没发出去。
  • 2 个 dropped:技术上有 crash,但维护者威胁模型里不算 —— 见下面"2 个 dropped 是怎么剩下来的"。

28 个里严重程度分布 14 High / 13 Medium / 1 Low,横跨从 jsoncpp DoS 到 libaom heap-overflow 这种走在公开 codec API 上的内存错误。这是 30 这个数字的内涵 —— 不是 raw findings count,是已经经过维护者眼球的 count。

三个有代表性的案例

案例 1:openscreen 三个姊妹 jsoncpp DoS,一个 CL 同时修

前面 LG 的例子用的就是 openscreen。三个 LG —— receiver_message / sender_message / answer_messages —— 在同一周内各自独立跑出了形态相似的 crash:都是 jsoncpp 的 Json::Value::operator[] 在非 object Value 上被 string key 索引,触发 JSON_FAIL_MESSAGE 里的 abort()。

根因都是同一个 pattern:openscreen 的解析函数里写的是 

if (!value) {
  return Error(Error::Code::kJsonParseError, "Invalid message body");
}
// ... value[kResult] / value[kSequenceNumber] ...

而 operator! 只对 nullValue 返回 true。任何 arrayValue / stringValue / 数值 / 布尔都会通过这个 guard,然后在下一行的 operator[](string) 里走进 find() 的 precondition,JSON_USE_EXCEPTION=0 + NDEBUG 下 assert 被消掉但 abort() 还在,Cast sender 进程当场死掉。

因为我们把这三个解析器切成了三个独立 LG,每个 LG 上的 crash 独立成一份 issue,上游(jo@google.com、dx@google.com)在第二天就接受并合并了一个 CL —— "[cast_streaming][security] Prevent JSON parsing aborts with non-objects" —— 同时把 answer_messages.cc / receiver_message.cc / sender_message.cc 三处的 if (!value) 升级成 if (!value.isObject())。

LG 切分让根因相同的 bug 在不同入口都能被独立触发,上游合并起来反而轻松。如果不分开,我们大概率只会发现其中一个,另两个还在线上。

案例 2:libaom AV1 Encoder Heap-Overflow —— 走在公开 API 上的真 bug

libaom/heap-overflow-av1-restore-layer-context 是这一波最干净的 high 级 finding 之一。LG 是"AV1 SVC encoder mid-stream reconfig",entry 是公开的 aom_codec_enc_init + aom_codec_encode + aom_codec_enc_config_set。

P3 检查在 LG discovery 阶段就确认这条入口是公开的;P2 协议研究阶段把"aom_codec_enc_config_set 在中途调用时哪些参数允许变化"摸清楚;P1 写完后跑 60 秒就抛出 heap-buffer-overflow。维护者看到的报告是 root cause + 公开 API repro + 一条 minimal 修复建议,而不是一份带着 harness 私有结构的 dump。这条已 fixed —— 走完 LG discovery → harness → crash → upstream merge 的完整链路。

案例 3:libwebp NULL Pointer Dereference —— 同日 confirm,2 天内 fix

libwebp/null-deref-WebPMuxAssemble —— 公开 API WebPMuxAssemble 在 mux edit 路径上对一个未检查的 internal pointer 解引用。LG entry 是 WebPMux*;core set 里有 muxedit.c 的整条 chunk 装配链。fuzzer 在第一晚跑出 crash,走完 verify 流程后发到 webmproject#497882857,Google libwebp 团队当天就 confirm,2 天内 fix-merged。从 issue 到代码合并不到 48 小时,是对反向四原则"每个 finding 都是真 公开 API 上的真 内存错误"的最强背书。

2 个 dropped 是怎么剩下来的

49 个真 bug、2 个 dropped。这个比例的直接原因不是 verify 阶段救场(verify 是下一篇的主题),而是 LG + 反向四原则在生成阶段就把绝大多数源码层 FP 干掉了 —— P1/P2/P3/P4 任一不通过都不进入下一阶段(每条违规对应什么含义,见 Part 1)。Stage 4 reach probe 命中失败 / run probe 触发的 crash 被归到 P1/P2 违规直接打回让 LLM 重写,这就是为什么源码层 FP 几乎为零。

剩下的 dropped 来自一个共同的失败模式 —— 库本身的威胁模型在维护者心里和我们理解的不一致。一个典型例子是 V8 的 uaf-v8-deserializer-unresolved-external-reference:走的是公开的 Isolate::New(deserialize_params),P3 检查通过(确实公开 API),P4 检查通过(确实在攻击面上、可达 unsafe ops),技术上也确实是 UAF。但 V8 团队在他们的安全 triaging policy 里明确说:依赖攻击者控制的 snapshot blob 不在 V8 威胁模型里。所以这条会被 dropped,即使源码层完全干净。

这种"技术真但威胁模型上无效"的 case 是四原则覆盖不到的 —— 它不是源码层的问题,是跨产品的安全合约层面的问题。这也就是下一篇要讲的 Security Proxy 要解决的事情:把每个库的 threat model 显式编码,在 file 之前再过一道。

为什么"两周 481 个 fuzzer"是关键数字

两个数字单看都不算特别 —— "两周" 不算快,481 也不算多。但放在一起看就特别 —— 这是质量保证下的速度。481 里的每个 fuzzer 都通过了 P1–P4 的源码层审查,每个都对应一个 LG 上的有意义攻击面,每个都跑过 reach probe + run probe。这才是 LLM-driven harness generation 应该追求的吞吐量定义:不是"一夜生成 5000 个 harness"那种 garbage in / garbage out,而是"每个都能直接 file 上游"的可交付吞吐。

和手工驱动的 fuzz 相比:

  • 手工写一个 harness 平均 1–2 天,人挑入口、读 protocol、写代码、验。481 个手工 ≈ 1.5 年人力。
  • 本套流水线平均一个 LG 端到端约 30 分钟(LG discovery 5 分钟 + protocol 5 分钟 + build/fix 10 分钟 + adversarial validation 10 分钟),跨多个并发 worker 跑,两周完成。
  • 源码层 FP 几乎为零(2 个 dropped 都是 threat-model 层面而非源码层差错),换言之 P1–P4 的硬卡接住了几乎所有"看着像 bug 但其实是 harness 错"的 case。

做 fuzz 的人都知道:harness 不是产物,bug 才是。但没有合格的 harness 就没有可信的 bug。把 LLM 的产能用在生成合格 harness 上,而不是生成 raw bug 报告上,是这次工作的核心结论。

下一篇:fuzz 跑出来 crash 之后怎么办

到目前为止讲的都是"在 harness 阶段就过滤掉错误"。但即使一个 P1–P4 干净的 harness,跑起来也会抛出大量 crash —— 一晚动辄上万。下一篇讲 verify 流程:四原则再用一次检查具体崩溃路径、2 步独立复现、再用 Security Proxy 跟维护者的 threat model 对齐。这是把 FP 率压到 <5% 的最后一道闸门。

四原则 × 2 步复现 × Security Proxy:从 24,000 个 crash 里挑出值得 file 的几个。
