问题:fuzzer 抛出来的 crash 大多数不值得 file

做过 fuzz 的人都遇到过这场面:一个 harness 跑了一夜,早上来看 ASan 抛出 24,000 个 crash 输入。其中绝大多数是同一个根因的 cluster siblings,真正的独立 signature 可能只有几十个。这几十里又只有一小部分是真的、上游会承认的漏洞。

这一步如果做不好,就会变成"自动化报告噪声"的批发商。所以即便上一篇里的 LG + 反向四原则在生成阶段就过滤掉了大量错误,fuzzer 抛出 crash 之后还是要做 verify,这是上报上游之前的最后一道闸门。

我们的 verify 流程有三层,从轻到重排好:四原则再审一遍具体的崩溃路径(便宜,不通过就立刻 drop)→ 两次独立复现(中等成本,排除 harness-only 的 crash)→ Security Proxy 跟维护者的威胁模型对齐(认知成本最高,但漏过会很贵)。

第一层:四原则,但只针对崩溃路径

为什么生成阶段已经过过一遍四原则,verify 阶段还要再审一遍?因为 harness 生成本身不是确定性流程。在静态分析数据齐全的 project 上,LG / 调用图 / API 协议都能被精确抽出来,生成质量很稳;但在静态分析覆盖不全的 project 上,流水线会 fallback 到由 LLM 做语义层面的推断 —— 这一步会引入随机性,不同 project 之间 harness 质量并不均一。换句话说,生成阶段的反向四原则挡掉了大多数源码层错误,但不能保证 100%。所以一旦 fuzzer 抛出 crash,我们要拿这条具体的 trace 再过一次四原则,做二次检查。

这一层的关键词是"具体路径"。在第一篇里,我们说四原则是 harness 整体的源码层正确性条件。在 verify 阶段,我们不再审整个 harness,只审把 crash 触发出来的那条调用路径。

操作上,sanitizer trace 里的那串栈是起点。从 LLVMFuzzerTestOneInput 开始往下读,逐帧问:

  • P1(harness 自身) —— 这条路径上的 harness 代码,有没有用未初始化变量?有没有 use-after-free?有没有忘记释放?fuzz bytes 真的流到那个崩溃位置了,还是被 harness 内部覆盖了?
  • P2(API 协议) —— 这条路径上调用的库 API,顺序对吗?对象状态对吗?参数在合法范围吗?有没有 set_params 走在 init_ex 之前那种典型错位?
  • P3(安全边界) —— 这条路径有没有从 extern 拽出来的 static 函数?有没有跳过公开入口的输入校验?判断标准不是"是否用了内部符号"(我们鼓励但不强制走公开 API),而是同样的 crash 能不能在只调公开 API 的 harness 上复现。能 → P3 通过;不能 → P3 失败,这条 crash 在攻击者从外面够不着的代码路径上。
  • P4(攻击面) —— 这条路径上的 fuzz 入口,在库的当前公开攻击面上吗?是不是某个 deprecated demo 的入口?

任何一条 P 不通过 —— 这个 crash 就是 harness artifact,直接 drop。我们不会让任何一个 P violation 的 crash 进到下一层 verify;这一关失败的 crash 写进 dropped_crashes 列表后就关掉。

这一步成本很低,基本是读源码。它的作用是在最早期就把"明显是 harness 自己问题"的 crash 截掉,后面两层不用浪费在显然的 noise 上。

第二层:两次独立复现

P1–P4 全部通过,只说明 harness 的代码看起来对。它不能保证 crash 在真实世界场景里也会触发 —— 一个 crash 只能在 harness 里复现,而在真实使用模式下永远不会发生,那它顶多是个学术 curio,不是漏洞。

所以我们要求两次独立的、互不依赖的复现。

复现 1:Harness 重放

把 crash 输入再喂给原 harness 跑一遍,在 ASan 下跑多次,要求每次都crash 在同一个 top stack frames 上。这一步排除两类 noise:

  • Flaky crash —— 由调度、随机数、未初始化内存的偶然位置触发的非确定性崩溃。这种 crash 在不同次复现时栈不一样,排除。
  • 非确定路径 —— harness 自身依赖的全局状态导致同一输入产生不同执行路径。栈不稳定就是不稳定的 fuzz signal,我们不报。

这一步虽然简单,但它解决了一个关键的诚信问题:报上去的 crash 维护者必须能复现。如果连我们自己重放都不稳定,上游一定会复现失败。

复现 2:Production 路径

这是关键。前面 harness 重放只是确认 fuzzer 的输出是稳定的,没排除 harness 自身把库逼到了"实际调用者永远不会触达的形状"。所以我们要求在 harness 之外、用真实使用模式把这个 crash 再触发一次。两条路径任选其一:

  • 路径 A:库自己的 CLI 工具,加 ASan 重新构建。很多库有官方 CLI(dwebp、flatc、vpxdec、xmllint、upx...),它们就是库的"production 模式"。如果 PoC 输入塞给 CLI 也能崩,这就是金标准 —— CLI 就是 production。
  • 路径 B:30 行左右的最小 C/C++ 程序,只用库的公开头文件。不引 harness 私有逻辑,不调任何内部符号,完全模拟一个普通 library 用户怎么调。如果这个迷你程序也崩,就证明真实使用者也会崩。

这一步是最重要的 FP 杀手 —— 大部分"看似真实但其实是 harness 在某种奇怪形状下逼出来"的 crash 都过不了这一关。两条路径都试过都不复现 → 这个 crash 是 harness-only,直接 drop。

我们 file 的每个 issue 都附带一个自洽的 repro/ 包(build.sh、run.sh、generate_poc.py、那个 30 行 reproducer 的源码、ASan log)。维护者从这个包就能 rebuild,不需要任何指向我们 repo 的链接。这件事看起来不起眼,但它把"维护者复现成本"降到了几乎为零,这是上游友好的核心。

第三层:Security Proxy —— 跟维护者的威胁模型对齐

过了前两层的 crash,技术上看是真的 —— harness 干净、复现稳定、production 路径也能崩。但这不代表它一定是漏洞。每个库都有它自己的安全契约,有些"看起来很可怕的 crash"在这个契约下属于"caller 应该自己防的事",上报上去会被关。

这一层处理的就是跨产品的安全合约层面的判断 —— 四原则覆盖不到、复现也覆盖不到的那种 FP。

每个库一份 security_proxy.yaml

我们对每个 fuzz 的库都维护一份 security_proxy.yaml,明确写下:

  • 哪些输入路径维护者认为是 "trusted",哪些是 "attacker-controlled"
  • 哪些 API 是公开攻击面(in-scope),哪些只是 caller 内部使用(out-of-scope)
  • 哪些 crash class 维护者明确不会当作漏洞处理
  • 已知的、维护者已经在 SECURITY.md 里声明过的 caller-contract

最关键的硬性要求:每条都必须 cite 上游文档,格式是 source: DOC:<path>,引用必须落到上游仓库里的具体文件 —— SECURITY.md、公开头文件注释、官方 README、维护者在 issue tracker 里的明确表态。

不允许我们自己的解读。如果某条规则我们想加但找不到上游 DOC 来源,这条规则不进 proxy。这种"严格保守 + 必须有引文"的原则是反对错填 proxy 把真 bug 错误地排除掉的护栏。

三个真实例子,看 proxy 怎么救你

例子 1:flatbuffers —— 不跑 Verifier 的 crash 都不算

flatbuffers 维护者在文档和 CppUsage 指南里反复强调:你的代码必须先调 VerifyBuffer,再访问数据。如果你跳过 Verifier 直接 GetRoot<T> 然后去读 fields,这是 caller-contract 违规,任何 crash 都不是库 bug。

这就是为什么我们对 flatbuffers fuzz 的每个 LG 都会先在 proxy 里看一遍:这个 LG 走 Verifier 吗?如果不走,fuzzer 跑出再像 OOB 的 crash 也不能 file。从这一波 fuzz 的 8 个 flatbuffers findings 里,有不少候选 crash 在 proxy 这一层就被识别成 "需要先 verify"——补上 VerifyBuffer 之后还能 crash 才报。

例子 2:libxml2 —— 库根本不打算信任输入

libxml2 的姿态在 SECURITY.md 里说得相当直接:它无条件信任它解析的 XML。"Don't feed it untrusted input" 是这个库的官方建议。所以 parser 路径上很多 OOM、unbounded recursion 之类的 crash 在维护者眼里都属于 "you fed it garbage",不属于 vulnerability。

没有 proxy 就会犯一个非常容易的错误:看到 sanitizer 抛 OOM 立刻 file,然后被维护者一句 "WONTFIX, see SECURITY.md" 关掉。账号信誉就慢慢被这种正确但无效的报告蚕食。proxy 让我们在 file 之前看一眼那条引用,如果根因落在这个 trust boundary 上,直接归到 dropped 里,理由记得清清楚楚。

例子 3:Dawn (WebGPU) —— BlobCache 不是 trust boundary

Dawn 的运行时 API 是强 trust boundary —— 浏览器层会校验、parser 内部也会再校。但磁盘上的 BlobCache 不是。Dawn 的设计文档明确说 BlobCache 是 integrity check(防止意外损坏),不是 authenticity check(防止恶意篡改)。

意思是:如果你的 fuzz 路径只能通过修改 BlobCache 文件触发 crash —— 那 Dawn 维护者会把它判 WontFix。攻击者可以写到那个文件,就早就有更直接的方法搞这台机器了。proxy 把这件事编码下来,你就不会拿这种类型的 crash 去打扰他们。

没 proxy 的实际代价

这一波 fuzz 里我们撞过一次 proxy 没覆盖到的边界 —— V8 的 uaf-v8-deserializer-unresolved-external-reference。技术上是真的 UAF,P1–P4 干净、harness 重放和 production 复现都通过。但 V8 团队在他们的安全 triaging policy 里明确把"依赖攻击者控制的 snapshot blob"列为 out-of-scope。所以 file 之后被标 Won't Fix,变成我们这一轮 3 个 FP 之一。

要是 proxy 在 file 之前就编码了 V8 的这条 policy(我们那时候还没维护 V8 的 proxy),这一报本来可以直接归进 dropped。这种丢分让我们更确信 proxy 的工程价值 —— 它的目的不是少找 bug,而是不浪费维护者的时间。

为什么这套流程值得做

从纯效率看,verify 流程对每个 crash 加了 5–60 分钟的成本;对一晚出来几万 crash 的 fuzz 而言,这是巨大的吞吐量损失。但损失的不是真 bug,损失的是 noise —— 三层闸门各自的目的就是更早把 noise 砍掉。

做对了的好处会延迟显现:维护者第一次见你时,看到的是一份带 root cause、带公开 API repro、带 fix patch 的报告。第二次再见,就会主动 ack。三五次之后,他们会愿意把你写的 harness 直接接进自己的 OSS-Fuzz 任务 —— 这是第一篇里 OpenSSL quic_server、fwupd 那 4 个团队的故事。

没做的代价同样会延迟显现 —— 你不会立刻看到"信誉损失",但慢慢地你的 GitHub handle 会被某些维护者 mute,你的 issue 默认低优,你的 PR 不被 review。这就是 LLM-driven fuzz 工作给整个 ecosystem 留下的心理债务。

把 verify 三层闸门都跑过的另一个隐性收益:你知道哪些 crash 你没有 file 以及为什么。这件事在被 maintainer 反问 "为什么你们没报 X?" 的时候非常有用 —— 我们能立刻拿出 dropped_crashes 的记录给到原因。这种透明度也是 ecosystem trust 的一部分。

还在思考的问题

这套流程不是终点。我们目前最关心的几件事:

  • 自动生成 proxy。每个新库手工写一份 security_proxy.yaml 是有成本的。我们正在尝试从 SECURITY.md / 公开头文件 / README 自动抽出条款,但抽得对不对、引文落得准不准还需要人审一遍。
  • Proxy 衰减。上游的 threat model 会随时间变化(新的 sandbox、新的 trusted boundary),proxy 没有 staleness 信号。我们得想出怎么在维护者更新 docs 时主动 invalidate proxy。
  • "无 proxy"的库。很多项目根本没发布 threat model。我们目前默认走最严格的策略(只报无可争议的内存安全 bug),但这是个 bet —— 我们可能错过了一些项目维护者愿意修的边界 case。
  • 二阶效应。我们最担心的不是 FP 灌水,而是真 bug 被维护者一并 auto-close —— 维护者看多了低质量 LLM 报告,看到自动化提交就先打 low-quality 标签。这是整个 ecosystem 层面的信任滑坡,单靠一个团队的 verify 流程拦不住。这件事我们最希望和 OpenSSF working group 一起讨论。

三篇串联起来:从 harness 审计到 vulnerability 上报

到这里,三篇 blog 完整地把四原则用了三次:

  • 第一篇 —— 用四原则审已有 harness,在 OSS-Fuzz 上拦下 14 个 FP、提了 53 个修复 PR、顺便挖出 25 年的 OpenSSL 老 bug。
  • 第二篇 —— Adversarial harness generation on the four principles:用四原则反向(P4 → P1)配合 Logic Group 切分,两周为 Chromium 及其依赖部署 472 个 harness,30 个 file 给上游(16 acked so far — 9 confirmed + 7 fixed),52 个在我们 audit 阶段就 drop 掉了。
  • 本篇 —— 用四原则第三次审具体崩溃路径,加上两次独立复现和 Security Proxy,把 file 上去的每个 issue 都做到对维护者真正友好。

一条规则能在审计、生成、verify 三个阶段都好用,说明它选的位置是对的 —— 在源码层做正确性的定义,而不是在 fuzz 输出后做事后挑选。这是这一系列工作的核心结论:harness 质量是 fuzzing 有效性的瓶颈;靠在生成器里把源码层检查内化进去解决,而不是靠 verify 阶段救场。
