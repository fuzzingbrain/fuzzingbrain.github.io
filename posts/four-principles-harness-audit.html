Harnesses Have Bugs Too: A Four-Principles Audit of 586 OSS-Fuzz Harnesses

Before we can ask an LLM to generate "good" fuzz harnesses, we need to answer a more basic question: what makes a harness good, and is there a checkable definition? This post offers our answer — the Four Principles — and validates it by applying the criteria to 586 production OSS-Fuzz harnesses across 70 projects. The audit prevented 14 false-positive crash reports from ever reaching upstream, produced 53 harness-fix PRs of which 45 (85%) were verified or merged by maintainers, and surfaced two real upstream bugs that broken harnesses had been masking for years — including one latent in OpenSSL's DES implementation for over 25 years.

从一个被绕开的问题说起

"I want an LLM to generate fuzz harnesses for me" — this goal has been getting heavy attention from both academia and industry over the past couple of years. But look one layer deeper and it rests on a premise almost everyone skips over: what makes a harness "good"? Without that definition, the only way to score what an LLM produces is "does it build" and "does it survive 60 seconds without crashing" — both necessary, neither sufficient. Prior work has measured FP rates as high as 94%.

Open-source projects on GitHub are increasingly being audited by AI tools at scale. Fuzzing has an outsized share of that attention because it ships with its own oracle in the sanitizer. A crash report with a sanitizer trace looks like ground truth, which makes fuzzing the default vehicle for AI-driven bug discovery and reproduction. The trouble shows up one layer down. When the harness itself has bugs, the fuzzer still produces a convincing-looking report. There is a sanitizer trace, a plausible call stack, the whole package. But the trace points at a problem no real caller can reproduce. The result is corrupted signal. Maintainers end up spending their attention separating real library bugs from harness-induced noise, and the "automation" and "scale" that AI research likes to claim quietly gets shifted downstream as triage burden.

That reduces the core question to a specification problem. Write down a source-level, checkable definition of what counts as a "good" fuzz harness.

Our answer is the Four Principles. The next section walks through them.

A definition by itself isn't enough. We also need to know whether those criteria actually capture "good", and the test we use for that is simple. Apply the criteria to a large sample of production harnesses, and check whether the ones the criteria flag are also the ones real maintainers want fixed. If maintainers consistently accept the resulting repair PRs, the criteria are tracking something real. If they shrug those PRs off, the criteria are noise.

So identifying bad harnesses isn't a downstream application of the Four Principles. It's the validation experiment for them.

We ran that experiment on OSS-Fuzz across 70 projects and 586 production harnesses. Here is what came back.

  • 53 violations identified. Each became a fix PR filed upstream.
  • 45 of those 53 (85%) were verified or merged by maintainers. 35 have already landed; the other 10 were acknowledged as real problems with the PR still in queue.
  • 14 of the 53 would have produced false-positive ASan or LSan crashes if left in place. The audit caught all 14 before they ever reached upstream.
  • Two latent library bugs surfaced once the broken harnesses were fixed, including a 25-year-old OpenSSL DES stack-buffer over-read.

The rest of this post follows that order. First the criteria themselves, then what the audit found in production, then the two latent bugs the audit unmasked.


The Four Principles: a source-level definition of a "correct" harness

To be accepted, a harness must pass four source-level checks at the same time. We call them the Four Principles (P1–P4). Each one targets one of the most common failure modes when an LLM (or a human) writes a harness.

P1
Logic Correctness
No bugs in the harness code itself
P2
API Protocol Compliance
Call order, object lifetimes, valid parameters
P3
Security Boundary Respect
Public APIs preferred · reproducible from the public side
P4
Entry Point Adequacy
Target sits on a real attack surface

P1 — Logic Correctness: no bugs in the harness itself

This is the simplest of the four principles and the one most often violated. A harness is a C/C++ program, and it makes the same mistakes any C/C++ program makes.

  • Variables initialized before use, pointers null-checked before deref.
  • Every error path releases its resources. No leaks, no double-frees, no use-after-free inside the harness.
  • Fuzz bytes actually reach the library API arguments. The harness is not a no-op, and it does not silently swap the fuzz input for hard-coded constants.
  • No static or global state leaks across LLVMFuzzerTestOneInput invocations.
  • The buffer sizes passed to API calls match the buffer sizes declared in the harness.

A P1 violation means the root cause is the harness itself, but the sanitizer trace will not necessarily stop inside the harness. The more common pattern is the harness pushing bad state into the library. Examples include oversized buffers, NULL pointers, uninitialized structs, and freed objects. The library code is what trips ASan or LSan, the trace looks like a library bug, and the root cause is still the caller violating the contract. That is the most confusing kind of P1 false positive. Wherever the trace lands, if P1 doesn't pass, the report is noise.

P2 — API Protocol Compliance: the harness must obey the library's contract

Every library API comes with its own usage contract. The contract covers when to init, when to destroy, what parameter ranges are valid, which return values must be checked, and which calls must be paired. LLM-generated harnesses violate these contracts a lot, because the LLM cannot infer a full state machine from a single function signature. The eight P2 sub-checks are listed in the P1.1–P2.8 checklist below.

Here is a real example. The libyaml_emitter_fuzzer on OSS-Fuzz is meant to do a round-trip check. Parse the fuzz input into a sequence of YAML events, push them through the emitter to get a serialized output, then parse that output back and confirm the event sequence matches. The simplified main control flow looks like this. 

int LLVMFuzzerTestOneInput(const uint8_t *data, size_t size) {
    yaml_parser_t parser;
    yaml_emitter_t emitter;
    yaml_event_t event;
    yaml_event_t events[MAX_EVENTS];
    size_t event_number = 0;

    yaml_parser_initialize(&parser);
    yaml_parser_set_input_string(&parser, data, size);
    yaml_emitter_initialize(&emitter);

    yaml_output_buffer_t out = { NULL, 0, 1000 };
    yaml_emitter_set_output(&emitter, yaml_write_handler, &out);

    /* ---- Loop 1: parse, save event, emit ---- */
    bool done = false;
    while (!done) {
        if (!yaml_parser_parse(&parser, &event)) goto delete_parser;
        done = (event.type == YAML_STREAM_END_EVENT);

        if (copy_event(&events[event_number++], &event)) {  // BUG
            yaml_event_delete(&event);                       //   returns 1 = success,
            goto delete_parser;                              //   harness reads it as failure
        }

        yaml_emitter_emit(&emitter, &event);   // never actually reached
    }

    /* ---- Loop 2: re-parse the output and compare events ---- */
    /* (entire loop is dead code: Loop 1 always exits at BUG) */
    yaml_parser_set_input_string(&parser, out.buf, out.size);
    int count = 0;
    while (!done) {
        if (!yaml_parser_parse(&parser, &event)) break;
        if (events_equal(events + count, &event)) break;
        yaml_event_delete(&event);
        count++;
    }

delete_parser:
    yaml_parser_delete(&parser);
    yaml_emitter_delete(&emitter);
    for (int k = 0; k < event_number; k++) yaml_event_delete(events + k);
    free(out.buf);
    return 0;
}

What's wrong: line 21 inverts the check. The helper copy_event() follows the libyaml convention of returning 1 on success and 0 on failure. The harness reads 1 as failure, so every successful copy jumps straight into the error-cleanup goto.

Why it's wrong: for any valid YAML input the fuzzer short-circuits on the very first event. The yaml_emitter_emit call inside Loop 1 never runs. All of Loop 2 (re-parse plus the events_equal consistency check) is unreachable. Roughly 50% of the harness logic has never been executed. The harness still compiles, still runs without crashing, and coverage still climbs (in the 50% that does run), which is why it sat on OSS-Fuzz for years without anyone noticing.

The fix is one character: add a !. if (copy_event(...)) becomes if (!copy_event(...)). After the patch landed, edge coverage jumped from 166 to 2479, a +1393% gain (about 15×). This is a textbook P2.4 (Return value handling) violation, with the harness mistaking success for failure. The full fix PR is at google/oss-fuzz#15096, merged.

A P2 violation means the crash is API misuse, not a library vulnerability. But P2 has a stealthier failure mode than just filing FPs, and the example above is exactly it. The harness compiles, runs, and gains coverage, but a wrong check has silently cleared the target path. The fuzzer looks busy. None of the sanitizer-relevant code on the actual target ever runs. You aren't failing to find bugs. You aren't testing in the first place. The sanitizer reports no crash, just a fuzz cycle that produces zero output. Worse than an FP.

P3 — Security Boundary Respect:鼓励走公开 API

真实攻击者通常只能从公开导出的 API 进。我们鼓励 harness 也尊重这条边界 —— 通过 extern 把 static 函数拽出来直接调,或者跳过公开入口的输入校验直接喂内部 chunk handler,会让 fuzzer 进到没人能从外面到达的代码路径,产生的 crash 难以对应到真实威胁模型。

P3 不是一刀切的"禁止用内部符号"。如果你 100% 确认这个内部用法是合理的(例如该内部状态机的输入语义和某个公开 API 的中间状态完全等价),或者能证明同样的 crash 也可以从公开 API 复现,这种 case 是允许的。判断标准是:能不能用一个只调公开 API 的 harness 把同一个 crash 重现出来 —— 能,P3 通过;不能,P3 失败。

  • 鼓励的写法:调公开头文件里声明的函数,不绕过公开入口的输入校验 —— 校验本身是库交付物的一部分
  • 谨慎使用内部符号:若 harness 必须用 static / friend / 内部 chunk handler,需要在 PR 里明确说明合理性,并附上"同样行为能在公开侧复现"的证据
  • 反例:直接调 png_handle_IHDR 触发 crash,但任何 PNG 文件都构造不出对应内部状态 —— 这是 P3 失败,因为公开侧无法复现

P3 是判断题,不是机械题。"某个 static 函数能不能合理拉出来用 / 同一个 crash 公开侧能不能复现"这种问题要靠项目语境 —— 资深 reviewer 读过库的 SECURITY.md、写过类似 fuzzer、了解项目历史,凭经验就能判;对没有项目上下文的第三方通用 LLM 来说不行,它没读过 SECURITY.md、每次给的理由都不一样,容易"看着像内部符号"就一律否掉合理用法。所以 P3(以及下面的 P4)在我们的流水线里都不让 LLM 单点拍板:LLM 列候选,静态调用图给可达性,引用必须落到 file:line,资深 reviewer review borderline。LLM 在这里是工具,不是裁判。

P3 违反 ⇒ 这个 crash 在没人能从公开侧到达的代码路径上。哪怕真崩了也很难修 —— 上游通常会标 NOT_REPRODUCIBLE_FROM_PUBLIC_API。

P4 — Entry Point Adequacy:目标必须在真实攻击面上

P4 决定的是:你选的这个 fuzz 入口,值不值得被 fuzz?一个把所有 fuzz 字节喂给某个 logging helper 或者 trivial accessor 的 harness 即使 P1–P3 全过,也是浪费 CPU。

  • 入口必须真的能 reach 到目标核心实现 —— 不是被早期 return 掉
  • 入口的可达闭包里必须存在内存安全相关的操作(memcpy、malloc、free、strcpy、sprintf 等),否则即使触发也无害
  • 目标本身在库的当前公开攻击面上,不是被 deprecated 的 demo 函数

P4 违反 ⇒ 这个 fuzzer 即使一直跑下去也不会找到有用的 bug。

P1.1–P2.8:16 条可勾选的子检查

把 P1 和 P2 落到工程上,我们各拆成 8 个具体的子检查。审计 / 生成 / verify 三个阶段都用同一套清单,这样三个阶段的判定才能互相对照。

P1 — Logic Correctness · 8 sub-checks

ID Sub-check What it requires
P1.1Resource leaksEvery exit path releases all alloc / fd / lock / handle
P1.2Use-after-freeNo read or pass of a pointer after free
P1.3Stale stateNo static / global state carried across LLVMFuzzerTestOneInput invocations
P1.4Input flowFuzz bytes actually reach the target API arguments (call not driven by constants)
P1.5Buffer safetyBounded fuzz-buffer reads, null-terminated C strings, length-checked indexing
P1.6Size checksEarly return when input is below the API minimum
P1.7Undefined behaviourNo OOB / null-deref / signed overflow / unaligned cast
P1.8No reimplementationCalls real library code, not an inline local copy

P2 — API Protocol Compliance · 8 sub-checks

ID Sub-check What it requires
P2.1Init sequenceAll required predecessors called in order before the target
P2.2Parameter constructionEach parameter has the right type, range, owner, and lifetime
P2.3Object lifecycleOpaque objects go through create → configure → use → destroy
P2.4Return value handlingReturn codes checked, error branches exercised, gated output not reused after failure
P2.5Cleanup sequenceResources released in API-documented order on every exit path
P2.6API existenceEvery called function is exported in the pinned build of the library
P2.7Co-call constraintsPaired APIs co-occur; mutually exclusive APIs do not
P2.8Prerequisite stateExternal state (fd / socket / env / thread) set up before the target call and torn down after

Implementation details (which sub-checks use LLM-read source with file:line citations, which use GDB breakpoints on the target API, which read ASan / LSan sanitizer traces) and how prober inputs differ across the three stages are covered in the next post on the reverse pipeline. This section just lays down the criteria themselves.

审计:70 个项目 × 586 个 harness

理论说完了,真要看四原则有没有用,得拉到生产代码上跑一轮。我们选了 OSS-Fuzz 上 70 个 C/C++ 项目作为审计对象 —— 这些项目的 harness 大多由项目维护者本人或资深贡献者写,经过上游 PR review,在 ClusterFuzz 上跑了好多年,被普遍认为是"高质量基线"。

从这 70 个项目里我们抓出了 586 个生产 harness。审计流程是:

  • 静态过:LLM agent 直接读 harness 源 + 库头文件,跑过 P1.1–P1.8、P2.1–P2.8 全部 16 个子检查 —— 引用必须落到 file:line
  • 动态过:对可疑点构造对抗性输入(adversarial probing),跑 60s libFuzzer + ASan/LSan,看是否真的触发
  • 对每个 violation 提交"修好的版本",并跑同样 60s 看覆盖率不降 —— 给上游一个直接 merge 的理由
  • 每个 harness 平均 10 分钟,10 个并发,总成本 ≈ $720

然后看结果。

53 个 violation,跨 28 个项目

我们一共找到 53 个 P1/P2 violation,占 586 个 harness 的 9.0%,分布在 28 个项目里。剩下的 42 个项目,这一轮我们没找出违规。需要说明的是,四原则不是确定性的判定 —— "没扫出问题"不等于"项目一定干净",只能说在我们这次审计中,这些项目的 harness 没触发任何子检查。这一点边界,值得提醒一下。

53 个全部以 PR / issue 形式提交给了对应上游。截至发稿:

14 个永远不会变成噪声 bug 报告的 FP

53 个 violation 里有 14 个会直接产生 FP 级别的 ASan/LSan crash —— 也就是说,如果我们绕开审计阶段、直接拿这些 harness 去 fuzz,这 14 个 harness 就会冒出"看似真实"的 crash,被自动化流水线打包成 14 份 bug report 砸到上游。

这 14 份报告里没有一个是真 bug。它们的根因都在 harness 里 —— 一个 dict 没 free、一个对象生命周期错位、一个内部 chunk 解析直接被驱动。每一份都会让维护者花时间读 stack trace、构建 repro 环境、对照源码、最后失望地 close 为 NO_VULNERABILITY,顺便给我们的提交账号留下"质量低"的印象。

四原则把这 14 个全部拦在了 fuzz 阶段之前。它们不是"跑出来再被 verify 排除",而是连跑都没跑就被静态识别 + 修复 + 提 PR 的方式消化掉。涉及的项目包括:

  • lcms — dict 和 transform_ext 两个 harness 都有 P2 违规导致的 FP crash;修完以后干净
  • libarchive / linkify — P1 资源生命周期问题
  • libpng / readapi — P2 调用顺序错位
  • njs / script — P1 静态状态没重置
  • openvpn — packet_id、verify_cert 都有 P1 违规导致的假 leak
  • wamr / mutator — P2 协议违反

这是审计的最大杠杆:不让 FP 流出去,就不需要事后再去修复维护者对自动化报告的不信任。

意外战利品:两个被 harness 遮住的真上游 bug

做这种审计有个有意思的副作用 —— 修一个 P2 违规,有时会把库本来就有、但被错误的 harness 长期遮住的 bug暴露出来。这次审计里出了两个这样的案例。

案例 1:OpenSSL DES 栈缓冲读越界 —— 潜伏 25+ 年

OpenSSL 的 provider harness 在调用顺序上有一个微妙的错误:它把 EVP_CIPHER_CTX_set_params 放在了 EVP_EncryptInit_ex2 之前。这违反了 P2 —— 在 context 还没初始化的时候 set params 会直接 return 0,等于这条 cipher 路径永远是 dead code。

修好这个调用顺序之后,harness 的覆盖率立刻多了 +51.4% edges,fuzzer 跑了几秒就抛出了 DES_ede3_ofb64_encrypt 里的 stack-buffer-overread。根因是 num 参数没有上界检查,可以从 EVP API 设到 ≥8,使第一轮循环就读到栈本地 DES_cblock d[8] 之外。

这个 bug 的本体在 crypto/des/ofb64ede.c 里趟了 25 年以上 —— 是 OpenSSL 项目本身比 OpenSSL 这个名字还要老的代码。它没被发现的原因不是难找,而是过去所有走到这条路径的 harness 都因为 P2 错误根本没真正进到这个函数。一个调用顺序错误,把一个公开 API 的内存安全 bug 隐藏了 1/4 个世纪。

上游已经合并了我们的修复,issue 也关闭了。

案例 2:tidy-html5 内存泄漏 —— 缺一次 cleanup 调用

tidy-html5 的某个 harness 漏掉了一次必要的 cleanup 调用,违反 P2 的"paired API"约束。补上这一次调用之后,harness 立刻在原本被 cleanup 掩盖的路径上抛出 LeakSanitizer 警告 —— 是库内部 lexer 里的真 leak。

根因和上面那条 OpenSSL 完全是一类:错误的 harness 用法把库的真实行为掩盖了。你可以把这两个看作"P2-driven bug discovery"的代表 —— 修 harness 不只是把 FP 干掉,更是在把库的真实行为重新暴露给 sanitizer。

维护者反馈:不止是合并 PR

这次审计的结尾不是 35 个 PR 合进 OSS-Fuzz 然后散场(剩下 10 个维护者已 acknowledge 但 PR 还在排队)。有 4 个维护者团队进一步把我们的 harness 或衍生 artifact 集成进了他们自己的工程化流程:

  • OpenSSL 直接采纳了我们对 quic_server harness 的重写版本(P1+P2 双修复),分支覆盖率提到 2.64×。这个版本现在跑在 OSS-Fuzz 的 OpenSSL 任务里。
  • fwupd 在他们自己的 fuzz 集成里把我们的修复版当作起点,后面又在此基础上写了新的 harness。
  • 另两个团队把审计过程里产生的副产品(corpus、structured config)纳入了自己的回归集。

这是这次工作里最让我们意外的回报 —— 维护者不是把我们的 PR 当作"AI 自动化的低质量噪声"草草接受,而是看到了"高质量审计 + 可执行的修复"之后,主动反向把我们的 artifact 当作工程化资源。

维护者的反馈直接验证了开篇那个 58% 不回的猜测:他们不是反对自动化,他们是反对没经过质量把关的自动化。

顺带:覆盖率的提升不是目标,但很可观

必须澄清一件事:四原则审计的目的不是涨覆盖率,目的是修正 harness 的正确性。但在很多 case 里,修一个 P1/P2 违规会顺便把死代码激活,覆盖率会跟着涨上去。我们把这当成 PR 通过的"无害证据"附在每个修复里。一些代表性的数字:

  • opencv/filestorage — +986% edges(原 harness 几乎是死代码)
  • libyaml/emitter — 14× edges
  • openssl/quic_server — +65.7%
  • bzip2/bzip2_fd — +52.4%
  • openssl/provider — +51.4%(就是上面那个 25 年老 bug 的现场)
  • tidy-html5/general — +33.5%
  • boost/filesystem — +21.9%

大幅度的提升通常对应"原 harness 是 dead code"的极端 case;小幅度通常是 harness 已经在跑、但有一个非阻塞的 P1/P2 违规(比如 jq/parse_stream)。两种都是 harness 质量问题,只是表现不一样。

为什么这件事在 LLM 时代更重要

手写 harness 的时代,质量问题靠人审 PR 拦掉;LLM 时代,harness 可以一夜之间生成几百个,如果不预设静态质量门槛,自动化就把"speed"翻译成了"FP 流量"。

四原则不是"LLM-only"的工具 —— 它是源码层的 harness 正确性定义,既适用于人写的 harness 审计(就是这篇里 586 个的来源),也适用于 LLM 生成 harness 时的自检。我们在生成阶段也用同一套四原则,只是反过来用 —— 那是下一篇的内容。

如果你现在在做 LLM-driven fuzz harness generation,我们建议:

  1. 不要把 build 通过和 fuzz 60s 没崩当作"quality good";这两个信号都是必要不充分。
  2. 在生成完 harness 之后强制跑 P1.1–P1.8、P2.1–P2.8 的子检查,违规的让 LLM 重写,而不是放进 fuzz queue。
  3. P3、P4 在生成之前用,选目标的时候就过一遍;选错了再写代码是浪费。
  4. 给上游的每个 PR 必须附 root-cause + 修复 + 覆盖率不降的证据,不能只甩 sanitizer trace。

这套流程的成本是有限的:586 个 harness 平均每个 10 分钟、10 并发,总开销大约 $720。换来的是 14 个永远不会成为噪声报告的 FP、45 个被维护者验证或修复的 PR(其中 35 个已 merge)、以及两个被失效 harness 长期掩盖的真上游 bug(含 OpenSSL 潜伏 25 年的那条)。这笔账无论从工程化成本还是 ecosystem trust 角度都是正的。

下一篇:把四原则反过来用
