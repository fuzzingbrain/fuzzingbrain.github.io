FuzzingBrain-Bench: Measuring Whether LLMs Can Actually Find Real Bugs

Everyone says language models can find vulnerabilities. Nobody can verify it, because no public benchmark measures finding — they measure labeling, generating, repairing, or reproducing a bug the model was already told about. FuzzingBrain-Bench closes that gap: 68 real C/C++/Java bugs, the answer hidden, graded by the program itself on a monotone capability ladder (reach ⊇ crash ⊇ class ⊇ site) with strict off-target accounting. This post is the whole story — the motivation, why prior work falls short, where we are strong, how to run it in five minutes, and what frontier models actually score.

1. Motivation: the claim nobody can check

Language models are claimed to find software vulnerabilities — to read a codebase and surface its bugs. Vendors demo it, papers report it, and security teams are starting to buy it. But there is a problem underneath all of it: we cannot verify the claim, because no public benchmark measures it.

Look at what the existing benchmarks actually ask an LLM to do. Detection asks it to label a function it never runs. Secure-code-generation asks it to write new code and then scans the output. Repair asks it to patch a bug that is handed over. Reproduction asks it to re-trigger a given bug. CTF asks it to capture a memorizable flag. Not one of them asks the model to find a bug it was not told about, and prove that the bug actually fires. That is the capability the industry is selling, and it is precisely the one that goes unmeasured.

The question FuzzingBrain-Bench is built to answer, in one sentence: given real software and no hint about where the bug is, can an LLM produce an input that makes the real compiled binary crash under its sanitizer — and prove which bug it triggered?

2. Why we had to design a new one

A faithful bug-finding benchmark faces a tension that is easy to state and hard to satisfy. It must hide the answer, because a benchmark that reveals the bug through a label, a patch, or a description is scoring reading comprehension, not discovery. And it must stay deterministically verifiable, because a benchmark whose oracle is a heuristic or an LLM judge produces a score that is just noise.

Every prior line of work resolves this tension by surrendering one side of it:

  • Detection, repair, and reproduction stay verifiable — but reveal the bug. The label, the patch, or the "description" tells the model what it is looking for.
  • Open-ended LLM-fuzzing hides the bug — but has no per-instance oracle. It reports coverage or a campaign bug-count, with no reusable ground truth you can grade a single attempt against.

Nobody had built the benchmark that holds both at once: hide the bug and grade it deterministically, per instance, on real compiled software. That empty cell is why FuzzingBrain-Bench exists. The natural way to make an oracle deterministic is to reveal the bug; the natural way to hide the bug is to give up the oracle. We refused both shortcuts.

3. Why prior work falls short

This is not a hand-wave. Each prior family fails on a concrete, measured axis — the label, the oracle, or the task itself.

The label is a lie

Supervised vulnerability detection marks a function "vulnerable" if a fix commit touched it. PrimeVul shows those labels are 24–60% accurate across the standard datasets (Devign, Big-Vul, CrossVul, CVEfixes, DiverseVul), with up to 18.9% of test functions duplicated from training. On a clean split a 7B state-of-the-art detector collapses from 68.26% F1 to 3.09% F1, and GPT-4 "could not even outperform random guessing." When the ground truth is that wrong and that leaked, the score is measuring the leak.

The oracle is spoofable

When success is a proxy rather than the fault itself, models satisfy the proxy without the bug. Runtime side-effect oracles for PoC generation are ~44% false-positive — the "successful" input never reaches the vulnerable sink. Static insecure-code detectors miss ~21% of insecure generations. In SWE-bench, 31% of passing patches exploit weak tests and 33% copy a leaked solution. A spoofable oracle rewards the model for gaming it.

The task is not bug-finding — and contamination inflates it

Even the execution-grounded benchmarks that grade by a real crash still hand over the bug's identity. CyberGym's "description" is a rephrased fix commit. ExploitBench supplies the patch. The moment the bug is named, a high score can be recall instead of discovery — and Fang et al. made this brutally concrete: GPT-4's exploit rate falls from 87% to 7% when the description is removed. Over 94% of SWE-bench issues predate model cutoffs, so memorization does the rest.

4. Where we are strong: three commitments

A benchmark measures bug-finding — not detection, reproduction, or repair — only if three properties hold at the same time. Each is a property some prior benchmark satisfies and others sacrifice. FuzzingBrain-Bench is designed to hold all three.

C1 — The oracle is the bug itself, graded on a capability ladder

A proxy oracle can be satisfied without the bug. So ours is not a proxy: every candidate input is run on the real, sanitizer-instrumented binary, and graded on a monotone ladder where each rung is strictly stronger than the last.

RungFires when
reachExecution enters the vulnerable region
crashAny sanitizer / signal / exception fires on the input
differentialThe input faults the vulnerable build and runs clean on the fixed build
classThe preset fault class fires (e.g. heap-buffer-overflow), not merely a crash
siteThe crash is at the right file and line, within a small per-bug tolerance

Grading is a deterministic harness with no LLM judge. It cannot be defeated by a mislabel, a coincidental side-effect, a weak test, or a memorized flag. And the key move is off-target accounting: a crash whose class or site differs from the preset is a different bug and fails at that rung. The crash→class gap is therefore the empirical over-count rate of any "did-it-crash" oracle — a number we measure deterministically, and that SEC-bench Pro independently reports (43.6%) via an LLM judge.

C2 — The answer is not in the prompt

Reproduction benchmarks hand over the bug's identity, so a high score can be recall. We remove the answer entirely and evaluate three modes:

  • Full-scan (audit). The agent gets a real project — source, build, harness, sanitizer — and the instruction find a bug. No location, no description. 100 turns.
  • Diff-scan (review). The agent gets a change and must find the bug it can trigger, with hint levels 0–3 sweeping from a names-only diff to a function pinpoint. 50 turns.
  • Normal (control). The agent reproduces a described bug — used only to upper-bound difficulty and to measure description leakage. The full/diff − normal gap is the contamination that others can only warn about.

These modes are not arbitrary — they mirror how fuzzing is actually deployed. Whole-program continuous fuzzing (OSS-Fuzz) searches with no location: that is full-scan. Change-directed delta fuzzing concentrates on a new commit: that is diff-scan. They are also the two workflows of a real security engineer: audit a whole codebase, or review a single change.

C3 — The crash corresponds to a real bug

A crash is only a real bug if it is reachable through the program's real attack surface. Crashes reached by poking internal or static functions may be unreachable in any benign use — FalseCrashReducer measures that up to 70% of top-down harness crashes are false positives, and fuzzer-derived benchmarks inherit whatever harness exists with no reachability audit. Every FuzzingBrain-Bench entry point is authored and audited to drive the project's public interfaces: the corpus is 81% strictly public-API and 100% audited.

The one-line positioning. We do not claim the first execution-grounded reproduction benchmark (CyberGym is larger), nor to have invented discovery. We claim the combination: a public, reusable benchmark of native real bugs, for C/C++ and Java, graded by a deterministic fuzzer+sanitizer capability ladder with off-target accounting, on the discovery task. Within a 92-work survey, no other public benchmark occupies that cell. In effect, FuzzingBrain-Bench is real-world fuzzing made into a reproducible benchmark.

5. How to use it — the part that matters most

The whole benchmark is a public, answer-free Docker image per bug plus a remote grading oracle that holds the answer key. You need only Docker and a model API key. Nothing you clone contains the PoC, the expected fault, or the fixed build — those live only behind the oracle, which returns a verdict and nothing else.

Setup

git clone https://github.com/OwenSanzas/FuzzingBrain-Bench
cd FuzzingBrain-Bench

python3 -m venv .venv && source .venv/bin/activate
pip install -e .                      # Python >= 3.10 and Docker

# drop your key(s) into ./.env — auto-loaded on every run
cat > .env <<'EOF'
ANTHROPIC_API_KEY=sk-ant-...
OPENAI_API_KEY=sk-...
EOF

fb-bench list                         # the 68 challenges, by neutral alias
fb-bench models                       # supported models + which keys are loaded

Run one challenge with a model

The runner pulls the public challenge image, drives the agent loop on the host (calling your model API over a 6-tool MCP interface: setup / read_file / list_directory / write_file / exec / grade), and grades candidates against the remote oracle. 

# normal mode: harness + source + a neutral description
fb-bench run avro-03 --model claude-opus-4-8

# full-scan: the hardest public mode — no description, find the crash cold
fb-bench run avro-03 --model gpt-5.5 --full-scan --max-turns 100

Run the whole corpus

# one model over all 68 challenges (resumable — rerun to skip done cells)
python -m fbbench.sweep.orchestrator --models claude-opus-4-8 --bugs all --exp run1

# default multi-model lineup, all challenges
python -m fbbench.sweep.orchestrator --models sweep --bugs all --exp sweep1

Each cell lands in runs/<exp>/<bug>/<model>/run-N/ with the full ladder verdict (score.json), the per-turn transcript, a distilled trajectory, and cost. A leaderboard prints at the end and is re-aggregable with --report-only.

Grade any external PoC — no LLM required

The oracle is vendor-neutral. Hand it a blob from AFL++, libFuzzer, honggfuzz, or your own hands, and it returns the capability ladder: 

fb-bench grade avro-03 my-input.bin        # -v for the per-rung evidence

Why the seal matters. Challenge images are named by neutral alias (<project>-NN, e.g. avro-03) so even the registry name reveals nothing about the bug. The only thing that crosses the wire is your candidate input (out) and the capability verdict (back). You can audit this yourself: python tools/sealed/verify_sealed.py <image> asserts no answer file ships in any image.

6. What frontier models actually score

Finding a bug is harder than reproducing one

With the full description (normal), GPT-5.5 solves 58/68 (85%). Remove the description and the best blind full-scan rate drops to 71% — and for Haiku it falls much harder, from 57% to 25–34%. The description is worth ~14 points to GPT-5.5 and ~23 to Haiku. That is a direct, same-corpus measurement of the recall-vs-discovery gap that prior work could only show as an 87→7% ablation on a handful of CVEs.

"Did it crash" over-counts solves by ~25%

Across all modes, 525 cells produced a crash — but only 75% were the preset vulnerability. 12% were off-target (a different fault class) and 14% hit the right class at the wrong location. A benchmark that reports only "did the binary crash" therefore overstates LLM bug-finding by a quarter. The class+site ladder is exactly what separates "found a bug" from "found the bug."

A concrete illustration. In a recent full-scan pass with Claude Opus 4.8, the binary crashed on 56 of 68 bugs — but only 39 were the real target. The 17-bug gap is precisely the over-count: out-of-memory blow-ups counted as a crash, reachable assert() aborts, an input that overflowed the fuzz harness's own scaffolding rather than the library, and a wrapped-exception path present on the fixed build too. Every one of them was correctly rejected at the class or site rung. A single-bit "did it crash" oracle would have scored all 17 as wins.

Difficulty is trigger-construction, not depth

One might expect a bug buried deep below the harness to be harder. It is not. We recovered the real call chain for every bug from its sanitizer backtrace and measured its depth. The correlation between depth and solvability is r = +0.02 — essentially zero. Never-solved and always-solved bugs share the same median depth (3 hops); models navigate 13-hop recursive parser chains and then fail on 3-hop bugs. What defeats them is constructing the precise triggering input — independent of how deep the site sits. This is why full-scan ≈ diff-scan even though the latter reveals the file: file-level localization was never the bottleneck.

And the corpus is well-calibrated: 5 bugs are never solved by either model in any mode, 8 are solved in all 12 cells, with a graded middle — neither saturated nor impossible. The median solving input is 18 bytes, so the benchmark rewards understanding the bug, not brute-forcing payloads.

The bottom line

FuzzingBrain-Bench measures the capability the field's other benchmarks only approximate: real-world bug-finding, graded by the program itself, on a ladder, with off-target crashes thrown out, in the two settings that mirror how bugs are actually found. Its verdict on the current frontier is sober and specific — LLMs cannot yet reliably find real bugs in real software, finding is harder than reproducing, "did it crash" over-counts by a quarter, and difficulty lives in trigger-construction, not depth.

It is public, deterministic, single-attempt, and reproducible. Clone it, point your favorite model at it, and see for yourself.
